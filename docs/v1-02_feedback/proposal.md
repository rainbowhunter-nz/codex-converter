# Feedback Proposal

## Goal

Improve the converter output so generated files are easier to review, settings can optionally include local overrides, users can choose to ignore generated Codex files in Git, and generated files can be cleaned up safely.

## Verified Current Behavior

- `.claude/settings.local.json` is currently discovered but skipped.
- `.codex/config.toml` is currently generated as a minimal placeholder.
- The CLI report already uses a Rich table, but statuses are plain text.
- There is no CLI flag that updates `.gitignore`.
- There is no CLI verb that removes generated converter output.

## Proposed Changes

| Feedback                                          | Proposed change                                                                                                              |
| ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Converted TOML has very long single lines         | Add a TOML formatting step after serialization so nested tables, arrays, and multiline strings remain readable.              |
| Conversion report table should be colorful        | Add status-specific Rich styles to the report table.                                                                         |
| Add option to convert `settings.local.json`       | Add an opt-in flag that merges `.claude/settings.json` and `.claude/settings.local.json`, with local values taking priority. |
| Review output and improve it                      | Improve generated headers, report detail text, config output formatting, and README examples.                                |
| Add flag to put generated files into `.gitignore` | Add an opt-in flag that appends generated output paths to `.gitignore`.                                                      |
| Add a verb `clean` to clean generated files       | Add a `clean` command that removes files known to be generated by `codex-converter`.                                         |

## CLI Changes

Extend `convert` with two flags:

```text
codex-converter convert [SOURCE] [--output PATH] [--dry-run] [--force] [--include-local-settings] [--gitignore]
```

- `--include-local-settings`: read `.claude/settings.local.json` when present and merge it over `.claude/settings.json`.
- `--gitignore`: add generated top-level outputs to `.gitignore`.

Keep `settings.local.json` skipped by default because it is machine-local and may contain developer-specific preferences.

Add a new `clean` command:

```text
codex-converter clean [TARGET] [--dry-run] [--force]
```

- `TARGET`: project directory to clean. Defaults to the current directory.
- `--dry-run`: show generated files that would be removed without deleting them.
- `--force`: remove generated files even when the generated marker is missing.

By default, `clean` should only remove files that contain a `codex-converter` generated marker. This keeps the command from deleting user-authored Codex files by accident.

## Settings Merge

Use a deterministic recursive object merge:

1. Start with `.claude/settings.json` when it exists.
2. If `--include-local-settings` is set and `.claude/settings.local.json` exists, merge it over the base settings.
3. For matching object values, merge keys recursively.
4. For arrays, strings, booleans, numbers, and nulls, replace the base value with the local value.
5. Convert the resulting settings object into `.codex/config.toml`.
6. Report unsupported merged settings the same way unsupported base settings are reported today.

Assumption: local settings should have full priority because the feedback explicitly says `settings.local.json` has higher priority.

## TOML Formatting

The converter should avoid hard-to-review TOML such as very long inline arrays or inline tables.

Implementation options:

- Prefer standard TOML tables over inline tables when generating known config sections.
- For lists of strings, emit one item per line when the line would otherwise become long.
- Use 120 characters as the maximum target line length for generated TOML.
- Keep generated files valid by parsing the result with `tomllib` in tests.

If `tomli_w` cannot produce the desired readable layout directly, add a small formatting helper scoped to the generated data shapes instead of a broad TOML pretty-printer.

## Report Styling

Add a status-to-style mapping in `cli.py`:

| Status        | Style    |
| ------------- | -------- |
| `written`     | green    |
| `planned`     | cyan     |
| `convertible` | cyan     |
| `skipped`     | yellow   |
| `unsupported` | magenta  |
| `conflict`    | red      |
| `error`       | bold red |

The table should remain readable without color because terminal color can be disabled or captured in CI logs.

## Gitignore Behavior

When `--gitignore` is passed, append generated output paths to the output project's `.gitignore`:

```text
AGENTS.md
.agents/
.codex/
```

This should ignore all converter-generated files, not only files generated during that specific conversion run. For nested `CLAUDE.md` conversions, also add the generated nested `AGENTS.md` paths.

Rules:

- Do not duplicate existing entries.
- Preserve existing `.gitignore` content.
- Respect `--dry-run` by reporting planned `.gitignore` changes without writing.
- Respect `--force` only for generated target conflicts; `.gitignore` should be appended non-destructively.

## Clean Behavior

`clean` should remove generated converter outputs from the target project:

```text
AGENTS.md
nested AGENTS.md files generated from nested CLAUDE.md files
.agents/skills/<converted-skill>/
.codex/agents/<converted-agent>.toml
.codex/config.toml
```

Rules:

- Never remove source Claude files.
- Never remove files outside `TARGET`.
- Never remove user-scoped Claude files under `~/.claude`.
- By default, remove only files that contain or can be verified from a `codex-converter` generated marker.
- Remove empty generated directories after generated files are removed.
- Do not remove `.agents/`, `.agents/skills/`, `.codex/`, or `.codex/agents/` if they still contain non-generated files.
- `--dry-run` reports planned removals without deleting anything.
- `--force` may remove known generated target paths even if a marker is missing, but it must still stay inside `TARGET`.
- Remove `.gitignore` entries that were added by `--gitignore`, while preserving unrelated `.gitignore` content.

Assumption: `clean` should be conservative because Codex project files may have been edited or created manually after conversion.

## Output Review Improvements

Improve generated output in small, concrete ways:

- Make generated headers consistent and concise.
- Use clearer report details for default config creation versus merged settings conversion.
- Update README examples for `--include-local-settings` and `--gitignore`.
- Keep unsupported hooks, MCP servers, plugins, auth, trust, and marketplace settings unsupported unless a safe direct Codex mapping is verified.

## Implementation Plan

1. Add tests for local settings merge priority and default skip behavior.
2. Add a settings loading helper that returns base settings, optionally merged local settings, and report entries for invalid JSON.
3. Replace placeholder config generation with readable TOML serialization for supported settings.
4. Add TOML formatting tests that guard against very long generated lines.
5. Style report statuses in the Rich table.
6. Add `.gitignore` append behavior behind `--gitignore`.
7. Add `clean` discovery and removal logic with marker checks, dry-run support, and generated `.gitignore` entry removal.
8. Update README usage and unsupported/skipped notes.

## Acceptance Criteria

- `convert` skips `.claude/settings.local.json` unless `--include-local-settings` is used.
- With `--include-local-settings`, local settings override base settings in `.codex/config.toml`.
- Generated `config.toml` parses with `tomllib` and avoids long unreadable single-line structures.
- Generated TOML targets a maximum line length of 120 characters.
- Conversion report statuses are styled in terminals that support color.
- `--gitignore` appends generated paths without duplicate entries.
- `--dry-run --gitignore` reports the planned `.gitignore` update without modifying the file.
- `clean --dry-run` reports generated files that would be removed without deleting them.
- `clean` removes generated files with converter markers and preserves user-authored Codex files.
- `clean --force` removes known generated target paths inside the target project even when markers are missing.
- `clean` removes `.gitignore` entries previously added for generated files and preserves unrelated entries.
- Existing tests continue to pass, with new tests covering the feedback items.
